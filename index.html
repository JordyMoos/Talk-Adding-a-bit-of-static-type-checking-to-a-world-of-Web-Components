The way that it works is that you specify the behaviour of your API in a DSL like this
Here we just specify which "Interactions" need to happen and what "Events" they need or produce.
It is then to baker to determine that the Interactions happen in the correct order. Even in parallel if it can.
This gets rid of the most complex part of writing large workflows and that is the orchestration. You now just need to write the interactions independently which makes it much easier.

Elm

  • Usability
  • Performance
  • Robustness
Elm is a frontend language with focus on Usability, Easiness, Performance and Robustness.

Elm

  • Pure Functional language
Elm is a pure functional language. The functional means what you think it means and that is that you have functions everywhere.
A function is a function as in JavaScript. You might give it a name. But essentially you specify which arguments it need and it will give you something back
You do not have `Methods` in Elm. Like you have in JavaScript. Elm does not have objects and therefor you neither have Methods on objects.
The biggest win is that you never have to worry what `this` actually means. Because there is no `this`.
